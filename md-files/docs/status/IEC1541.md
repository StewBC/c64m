# IEC / 1541 status

## Current implementation

- VIA 6522 module implemented and unit-tested (Phase 1 complete).
- 1541 machine shell: CPU, VIAs, memory map, ROM loading, cycle-step (Phase 2 complete).
- IEC bus wiring (Phase 3 complete):
  - `c1541_update_iec_bus()` runs every cycle: serial VIA ($1800, `via1`) Port B output bits → `c64_set_iec_external_pull()` → combined bus state → serial VIA input bits + CA1.
  - CA1/ATN IRQ: ATN assertion feeds `via6522_set_ca1()`, which sets the serial VIA IFR bit 1 on the falling edge; the 1541 CPU IRQ fires via the existing callback.
  - Serial VIA PB4 ATN acknowledge is modeled: while the C64 asserts ATN, PB4 configured output-low makes the drive pull DATA low immediately, matching the 1541 serial attention handshake.
  - New accessors: `c64_get_iec_c64_pull()` (reads CIA #2 Port A without side effects), `c64_get_drive_slot()` (returns mounted D64 slot pointer).
  - New peek helper: `cia_peek_port_a_output()` — returns raw ORA masked by DDRA without triggering port-input callbacks.
- D64 sector access via ROM intercept (Phase 3 complete):
  - Intercept at `reed` ($F4CA in DOS 2.6: 325302-01 + 901229-06AA). Job queue model: `jobn` at $3F, `hdrs[]` at $06, buffers at $0300+N×$0100.
  - On success: fills 1541 RAM buffer from mounted D64 image, sets PC to `read40` ($F505); on error: sets A = $02, PC = `errr` ($F969).
  - Queued READ/SEARCH jobs are satisfied from the mounted D64 image, including ROM SEARCH jobs whose queued command byte keeps its low-bit flag set.
  - Inline `d64_sector_offset()` table avoids `tools/d64/` dependency.
  - Guard: job index ≥ 5 returns early (job 5 = command channel, never a READ).
- D64 sector WRITE via ROM intercept (Phase 4 complete):
  - WRITE job (`$90`, `& $78 == $10`) handled at the same job-dispatch window and
    physical-job fallback as READ. DOS fills the job buffer before queuing the
    job, so `c1541_copy_job_buffer_to_sector()` persists `$0300+N×$0100` back into
    the mounted image at the target sector via `c64_get_drive_slot_mut()`.
  - Writable gating: WRITE to a read-only slot writes nothing and returns the
    write-protect job status ($08 → DOS 26); success sets `slot->dirty` so the
    runtime flush persists the image to the host `.d64`. No file I/O in `machine/`.
  - Covers SAVE, sequential/relative file writes, and BAM/directory updates.
- DOS command & error channel (Phase 5 complete):
  - Scratch (`S0:`), rename (`R0:`), validate (`V0`), and the error/status channel
    (`OPEN 15,8,15`) all work via the real ROM with no dedicated code — they are
    ordinary directory/BAM READ/WRITE jobs satisfied by Phase 3 + Phase 4.
  - Format (`N0:name,id`) is handled by intercepting the DOS FORMT EXECUTE job
    (`$E0`, `& $78 == $60`) in `c1541_format_track()`: the target track's sectors
    are erased in the writable image and the job returns success, after which the
    ROM's DOS writes a fresh BAM + directory (name/ID) through normal WRITE jobs.
    Read-only mounts return write-protect. No `d64_image_format()` helper needed.
- KERNAL LOAD trap coexistence:
  - `emulate_1541` flag in `c64_config` enables the 1541 ROM path for raw IEC / drive-side code.
  - Threaded through: `app_options.emulate_1541` → `runtime_config.machine_config` → `c64_config` → `c64_set_config()`.
  - INI key: `[disk] emulate_1541` (only written to INI when true).
  - When `emulate_1541` is enabled and a 1541 ROM is loaded for device 8/9, the C64 KERNAL LOAD trap is disabled for that device. Standard KERNAL `LOAD` runs through the real C64 KERNAL IEC path and the real 1541 DOS ROM serial handlers.
  - When `emulate_1541` is disabled or no 1541 ROM is loaded, the D64-backed KERNAL LOAD trap remains active for compatibility, including `LOAD "$",8` and `LOAD "*",8`.
  - Regression coverage includes direct real-ROM `LOAD"*",8` from `GALENCIA.D64` and the runtime autorun path (`-a --disk 8=...`) with success checked by BASIC memory contents, not by host RUN logging.

## ROM

- 1541 ROM loaded from INI key `[roms] 1541` or auto-discovered as `1541.rom` in `"."`, `"rom"`, `"roms"` directories.
- ROM must be the 16 KB combined image (lower 8 KB = 325302-01, upper 8 KB = 901229-06AA).
- Absent ROM is non-fatal: falls back to KERNAL LOAD trap.

## Known limitations / deferred

- Real 1541 DOS sector writes are implemented (Phase 4, job-level WRITE
  intercept). Still deferred: track-level format and media-level write fidelity
  (VIA #1 head / GCR / rotation) and G64.
- DOS command channel (format/scratch/rename/validate) and the error/status
  channel work via the real ROM (Phase 5). Still deferred: media-level format
  fidelity (real GCR track layout / custom formats), cross-drive copy (`C`), and
  block/memory commands (`B-*`, `M-*`).
- Additional ROM variants (1571, fast loaders that modify disk-controller VIA motor/sync): not validated.
- Devices beyond 8 and 9: not implemented.
- Disk-controller VIA mechanics: stubbed (head stepper, motor, SYNC not emulated — ROM intercept approach bypasses them).
- NTSC 1541 clock (should be ~1.023 MHz vs PAL ~985 kHz): PAL 1:1 lockstep only for now.
