## Background and motivation

The existing KERNAL LOAD trap at `$FFD5` handles the standard `LOAD "*",8` call
that runs the Galencia bootstrap. After the bootstrap runs, Galencia (and many
other modern C64 titles) performs secondary loads using either the KERNAL IEC
routines or a custom fast loader. Neither path is currently handled: no IEC
device responds on the bus, the machine hangs in the KERNAL serial wait loop at
`$ED5A`.

The correct architectural solution — and the one chosen here — is to emulate a
real MOS 1541 disk drive: give it its own 6502 CPU instance running the genuine
1541 ROM, its own VIA peripherals, and connect it to the C64 via the existing
IEC bus model. Because the CPU core is fully callback-driven with no global
state (confirmed by `C64IECEVAL.md`), a second instance is safe to create
without any changes to the CPU core itself.

The 1541 runs in **lockstep** with the C64 on the same runtime thread. No
additional thread is introduced. No inter-thread synchronisation is needed for
the IEC bus. The 1541 is stepped once per C64 cycle inside
`c64_advance_one_cycle()`, immediately after the C64's own CPU, CIA, VIC-II,
and SID steps.

## Architecture constraints

All rules from `AGENTS.md` and `MASTER.md` apply:

- The 1541 emulator belongs in `machine/`. New files: `src/machine/c1541.c`,
  `src/machine/c1541.h`, `src/machine/via6522.c`, `src/machine/via6522.h`.
- `machine/` may only depend on `util/`. No SDL, no Nuklear, no runtime, no
  platform, no frontend.
- The 1541 may access `src/tools/d64/` **only if** the dependency direction
  rules permit it. They do not: `tools/` is a peer layer, not below `machine/`.
  D64 data must be accessed via the same inline approach used by the existing
  KERNAL trap — working directly from `c64_drive_slot.image_bytes` — or via a
  shared data structure passed by `c64_t`. Do not `#include` any `tools/d64/`
  header from `machine/`.
- The 1541 lives on the runtime/emulation thread only. The live `c1541_t`
  struct must never be touched from the UI thread.
- The 1541 is owned by `c64_t`. It is not a separate top-level object managed
  by the runtime. The runtime steps the C64; the C64 steps the 1541.
- Thread ownership of the IEC bus: both the C64 and the 1541 share
  `c64_t.iec_external_pull`. Since both run on the same thread in lockstep,
  no locking is required. Do not introduce locks or atomics for the IEC bus.

## 1541 hardware summary

The 1541 contains:

| Component | Details |
|---|---|
| CPU | MOS 6502 (plain, no 6510 I/O port) |
| RAM | 2 KB at `$0000–$07FF` (mirrored to `$0800–$0FFF`) |
| ROM | 16 KB at `$C000–$FFFF` (the 1541 DOS firmware) |
| VIA #1 | MOS 6522 at `$1800–$180F` — parallel port (stepper, R/W head) |
| VIA #2 | MOS 6522 at `$1C00–$1C0F` — serial IEC port + drive LED/motor |

For the purpose of this emulator, VIA #1 (the disk mechanics port) needs enough
implementation to not crash the 1541 ROM's initialisation sequence — the ROM
writes to VIA #1 on startup. It does not need to model the actual stepper motor
or read head. VIA #2 is the critical peripheral: it drives the IEC bus lines
(ATN, CLK, DATA) that communicate with the C64.

The 1541 ROM is **not** included in the repository. It must be loaded from a
file at runtime, exactly as the C64 ROM is loaded. The ROM file is the standard
`1541.rom` / `dos1541` image (16384 bytes). If the ROM is not present, device 8
falls back to the existing KERNAL LOAD trap behaviour.

## MOS 6522 VIA overview

The 6522 VIA has the following registers (base address `B`):

| Offset | Name | Function |
|---|---|---|
| `B+0` | ORB / IRB | Port B data output / input |
| `B+1` | ORA / IRA | Port A data output / input |
| `B+2` | DDRB | Port B direction (1=output) |
| `B+3` | DDRA | Port A direction (1=output) |
| `B+4` | T1C-L | Timer 1 counter low |
| `B+5` | T1C-H | Timer 1 counter high |
| `B+6` | T1L-L | Timer 1 latch low |
| `B+7` | T1L-H | Timer 1 latch high |
| `B+8` | T2C-L | Timer 2 counter low |
| `B+9` | T2C-H | Timer 2 counter high |
| `B+A` | SR | Shift register |
| `B+B` | ACR | Auxiliary control register |
| `B+C` | PCR | Peripheral control register |
| `B+D` | IFR | Interrupt flag register |
| `B+E` | IER | Interrupt enable register |
| `B+F` | ORA (no handshake) | Port A without handshake |

The 6522 is similar in spirit to the 6526 CIA but with important differences:
no TOD clock, no BCD alarm, no CNT cascade. Timers count down from a latch;
the ACR controls one-shot vs. free-run mode, timer 1 PB7 output, and shift
register source. The IFR/IER model mirrors the CIA ICR model.

### What the 1541 ROM actually uses

The ROM init sequence configures VIA #1 and VIA #2 and then enters the serial
bus attention handler loop. For the purposes of IEC communication, the
following VIA #2 behaviour must be correct:

- **Port A (ORA/IRA, DDRA):** The IEC bus lines.
  - Bit 0: DATA out (to bus)
  - Bit 1: CLK out (to bus)
  - Bit 2: DATA in (from bus) — input
  - Bit 3: CLK in (from bus) — input
  - Bit 4: ATN in (from bus) — input
  - Bit 5: (device address selection, tied to hardware jumper)
  - Bit 7: ATN acknowledge out
- **Port B (ORB/IRB, DDRB):** Drive LED and motor control (VIA #2 PB) and
  device-address hardware lines. For emulation purposes, writes to PB are
  accepted and ignored (no real motor or LED to drive).
- **Timer 1:** Used by the ROM for bit-timing in the serial protocol. Must
  count correctly.
- **Timer 2:** Used less frequently; must count correctly.
- **CA1 interrupt:** ATN line from the IEC bus triggers CA1. The ROM uses CA1
  as an NMI source to interrupt the current operation when ATN is asserted.
  This is critical: when the C64 asserts ATN, the 1541 must receive a CA1
  interrupt that vectors through `$FFFA` (NMI vector).
- **Shift register:** Used by some fast loaders but not by the standard ROM
  serial protocol. Implement as readable/writable registers; behaviour need
  not be cycle-accurate for Phase 1.

### What VIA #1 must do at minimum

The ROM writes to VIA #1 during init. A stub implementation that accepts all
writes without crashing and returns sane default values on reads is sufficient
for Phase 1. VIA #1 does not need to model the actual disk mechanics.

## `via6522` implementation

### Struct

```c
typedef struct via6522 {
    uint8_t ora;        // Port A output register
    uint8_t ira;        // Port A input register (latched or live)
    uint8_t ddra;       // Port A direction
    uint8_t orb;        // Port B output register
    uint8_t irb;        // Port B input register
    uint8_t ddrb;       // Port B direction
    uint16_t t1_counter;
    uint16_t t1_latch;
    uint16_t t2_counter;
    uint8_t  t2_latch_low;
    uint8_t  sr;        // shift register
    uint8_t  acr;       // auxiliary control
    uint8_t  pcr;       // peripheral control
    uint8_t  ifr;       // interrupt flags
    uint8_t  ier;       // interrupt enable
    int      t1_running;
    int      t2_running;
    int      t1_oneshot_done;
    // CA1/CA2/CB1/CB2 input state for edge detection
    uint8_t  ca1_last;
    uint8_t  cb1_last;
} via6522;
```

### Functions

```c
void    via6522_init(via6522 *v);
void    via6522_reset(via6522 *v);
uint8_t via6522_read(via6522 *v, uint8_t reg);
void    via6522_write(via6522 *v, uint8_t reg, uint8_t value);
void    via6522_step(via6522 *v);          // advance one cycle
int     via6522_irq_pending(via6522 *v);  // 1 if IFR & IER & 0x7F
// External pin inputs (called by c1541 to feed bus state into VIA)
void    via6522_set_ca1(via6522 *v, uint8_t level);  // ATN in on VIA #2
void    via6522_set_port_a_inputs(via6522 *v, uint8_t pins); // CLK/DATA in bits
```

### Timer behaviour

Both timers count down by 1 per `via6522_step()` call. When T1 reaches zero:
- Set `IFR` bit 6 (T1 timeout).
- If ACR bit 6 = 0 (one-shot): stop T1, set `t1_oneshot_done`.
- If ACR bit 6 = 1 (free-run): reload T1 from latch.
- If ACR bit 7 = 1 (PB7 output): toggle PB7.

T2 counts down by 1 per step (pulse-counting mode uses CB1, which is not needed
for IEC; implement only the interval timer mode). When T2 reaches zero: set
`IFR` bit 5.

### Interrupt model

`via6522_irq_pending()` returns 1 if `(ifr & ier & 0x7F) != 0`. The 1541 CPU
IRQ line is driven by VIA #1 IRQ pending OR VIA #2 IRQ pending. The 1541 NMI
line is driven by the CA1 interrupt on VIA #2 when ATN is asserted (see
below).

### CA1 / ATN interrupt

CA1 on VIA #2 is connected to the IEC ATN line. When ATN transitions from high
to low (C64 asserts ATN), CA1 fires. The PCR bit 0 controls edge polarity
(active low edge = ATN asserted).

When CA1 fires:
1. Set `IFR` bit 1 (CA1 flag).
2. If the 1541 ROM has configured this as an NMI source (which it does during
   init via PCR), signal an NMI to the 1541 CPU. In practice, the 1541 ROM
   configures CA1 as the ATN interrupt and the ROM's NMI vector (`$FFFA/$FFFB`)
   points to the ATN service routine.

Implement CA1 edge detection in `via6522_set_ca1()`: compare new level to
`ca1_last`, detect the configured active edge (from PCR bit 0), fire the
interrupt flag if the edge matches.

## `c1541` machine implementation

### Struct

```c
typedef struct c1541 {
    C6510    cpu;           // reused 6502 core (cpu.class = CPU_6502)
    via6522  via1;          // parallel port (disk mechanics stub)
    via6522  via2;          // serial IEC port
    uint8_t  ram[2048];     // 2 KB internal RAM
    uint8_t *rom;           // 16 KB ROM (heap, loaded from file)
    int      rom_loaded;    // 0 if ROM not present; drives fallback behaviour
    c64_t   *c64;           // back-pointer for IEC bus access only
    int      device_number; // 8 or 9
} c1541;
```

The `C6510` struct is reused directly (confirmed safe by `C64IECEVAL.md` Q1–Q3).
`cpu.class` must be set to `CPU_6502` by `c6510_init()`, which it already is
for the non-6510 path.

### Memory map

The 1541 bus read/write callbacks implement the following map:

| Address range | Maps to |
|---|---|
| `$0000–$07FF` | RAM (2 KB) |
| `$0800–$0FFF` | RAM mirror |
| `$1800–$180F` | VIA #1 registers |
| `$1C00–$1C0F` | VIA #2 registers |
| `$C000–$FFFF` | ROM (16 KB) |

Addresses not in the above ranges return `0xFF` on read and are silently ignored
on write. The 1541 ROM never accesses addresses outside these ranges during
normal operation.

### Bus callbacks

```c
static uint8_t c1541_bus_read(void *user, uint16_t addr) {
    c1541 *drive = (c1541 *)user;
    if (addr < 0x0800) return drive->ram[addr & 0x07FF];
    if (addr < 0x1000) return drive->ram[addr & 0x07FF]; // mirror
    if (addr >= 0x1800 && addr < 0x1810) return via6522_read(&drive->via1, addr & 0x0F);
    if (addr >= 0x1C00 && addr < 0x1C10) return via6522_read(&drive->via2, addr & 0x0F);
    if (addr >= 0xC000) return drive->rom[addr - 0xC000];
    return 0xFF;
}

static void c1541_bus_write(void *user, uint16_t addr, uint8_t value) {
    c1541 *drive = (c1541 *)user;
    if (addr < 0x0800) { drive->ram[addr & 0x07FF] = value; return; }
    if (addr < 0x1000) { drive->ram[addr & 0x07FF] = value; return; }
    if (addr >= 0x1800 && addr < 0x1810) { via6522_write(&drive->via1, addr & 0x0F, value); return; }
    if (addr >= 0x1C00 && addr < 0x1C10) { via6522_write(&drive->via2, addr & 0x0F, value); return; }
    // ROM and unmapped: ignore
}
```

Set these as `cpu.read = c1541_bus_read` and `cpu.write = c1541_bus_write` with
`cpu.user = drive` during `c1541_init()`.

### IEC bus integration

VIA #2 Port A drives the 1541's side of the IEC bus. The mapping is:

| VIA #2 Port A bit | IEC signal | Direction |
|---|---|---|
| Bit 0 | DATA out | Output (1541 → bus) |
| Bit 1 | CLK out | Output (1541 → bus) |
| Bit 2 | DATA in | Input (bus → 1541) |
| Bit 3 | CLK in | Input (bus → 1541) |
| Bit 4 | ATN in | Input (bus → 1541) |
| Bit 7 | ATN ack out | Output (1541 → bus) |

After each `via6522_step(&drive->via2)` call in `c1541_advance_one_cycle()`:

1. **Read VIA #2 output lines:** compute what the 1541 is asserting on the bus.
   - DATA out: VIA #2 Port A bit 0, masked by DDRA bit 0. If output and low → 1541 pulls DATA.
   - CLK out: VIA #2 Port A bit 1, masked by DDRA bit 1. If output and low → 1541 pulls CLK.
   - ATN ack: VIA #2 Port A bit 7, masked by DDRA bit 7. If output and low → 1541 pulls ATN.

2. **Update `c64_set_iec_external_pull()`** with the combined mask of lines the
   1541 is pulling low. Use the existing `C64_IEC_CLK`, `C64_IEC_DATA`,
   `C64_IEC_ATN` flags from `src/machine/c64.h`.

3. **Read the C64's IEC output state** from `c64_t` and feed it back into VIA
   #2's input pins:
   - Compute the combined bus state (C64 output ORed with 1541 output — open
     collector: any puller wins).
   - Feed CLK in (bus CLK low → VIA #2 Port A bit 3 = 0) via
     `via6522_set_port_a_inputs()`.
   - Feed DATA in (bus DATA low → VIA #2 Port A bit 2 = 0) via
     `via6522_set_port_a_inputs()`.
   - Feed ATN in (bus ATN low → VIA #2 Port A bit 4 = 0) via
     `via6522_set_port_a_inputs()`.

4. **Feed ATN to CA1:** call `via6522_set_ca1(&drive->via2, atn_level)`. If
   ATN just went low (C64 asserted ATN), CA1 fires, sets IFR bit 1, and
   triggers an NMI to the 1541 CPU (if the ROM has configured it — which it
   does after init).

The C64's current IEC output state can be read by inspecting `c64_t`'s CIA #2
Port A register and DDRA through whatever accessor is available, or by caching
the last-written bus state in a new field on `c64_t`. Use whichever approach
is cleaner given the current `c64_t` internals — do not add a new public
accessor if one already exists.

### Cycle step

```c
void c1541_advance_one_cycle(c1541 *drive) {
    if (!drive->rom_loaded) return;  // no ROM: skip entirely

    // Step VIAs first (they may set IRQ/NMI before CPU evaluates them)
    via6522_step(&drive->via1);
    via6522_step(&drive->via2);

    // Update IEC bus: read VIA #2 outputs, write to c64 iec_external_pull,
    // read C64 IEC state, feed back into VIA #2 inputs and CA1.
    c1541_update_iec_bus(drive);

    // Determine IRQ and NMI state
    int irq = via6522_irq_pending(&drive->via1) || via6522_irq_pending(&drive->via2);
    int nmi = /* CA1 NMI pending on VIA #2 */ drive->via2_nmi_pending;

    // Step the 6502 one cycle
    c6510_step(&drive->cpu, irq, nmi);
}
```

The exact signature of `c6510_step()` and how IRQ/NMI are passed must be
confirmed against the current source before use. Follow the same calling
convention used by `c64_advance_one_cycle()` for the C64's own CPU step.

### Lifecycle

```c
// Load ROM from file path. Returns 1 on success, 0 if file not found/wrong size.
int     c1541_load_rom(c1541 *drive, const char *path);

// Init struct; does not load ROM. Call c1541_load_rom() separately.
void    c1541_init(c1541 *drive, c64_t *c64, int device_number);

// Free ROM buffer; release any resources.

```

### ROM path

The 1541 ROM path follows the same convention as the C64 ROM path — loaded from
a configurable directory. The filename should be configurable (different 1541
ROM versions are named differently: `1541.rom`, `dos1541`, `1541-ii.rom`).
Follow the existing ROM loading pattern in `c64.c`.

If no 1541 ROM is present: `c1541.rom_loaded = 0`. In this case
`c1541_advance_one_cycle()` returns immediately and the machine falls back to
the existing KERNAL LOAD trap for device 8/9. The fallback must continue to
work correctly when the 1541 ROM is absent.

### KERNAL LOAD trap coexistence

When the 1541 ROM **is** loaded, the KERNAL LOAD trap at `$FFD5` should be
**disabled for devices 8 and 9**. The real 1541 handles all LOAD calls through
the IEC protocol; the trap would duplicate and confuse the bus activity.

When the 1541 ROM is **not** loaded, the KERNAL LOAD trap remains active as the
fallback. This is the existing behaviour and must not regress.

The trap enable/disable should be per-device: if `drive8.rom_loaded`, disable
the trap for device 8; if `drive9.rom_loaded`, disable the trap for device 9.

## D64 data access from the 1541

The 1541 ROM handles D64 file access entirely through its own 6502 execution —
it reads sectors via VIA #1 (the disk mechanics port) in the real hardware.
In the emulator, VIA #1 is a stub and the actual sector data must come from
the mounted D64 image.

The approach is to intercept the 1541 ROM's sector read requests at the VIA #1
level: when the ROM signals a track/sector request via VIA #1 (specific port
patterns or timer interactions), the emulator satisfies it from
`c64_drive_slot.image_bytes` directly.

**This is a significant implementation detail that requires reading the 1541 ROM
disassembly** to understand exactly how the ROM communicates with VIA #1 for
disk reads. The relevant ROM routines are the GCR decode and sector fetch
handlers. An alternative approach is to intercept at the level of specific ROM
addresses (patching known sector-read entry points), similar to how the C64
KERNAL trap works.

**The recommended approach for Phase 1:** intercept at specific well-known 1541
ROM addresses rather than fully emulating VIA #1 disk I/O. The ROM address for
the sector read entry point is well-documented (e.g., `$D586` in the standard
1541 ROM). When the 1541 CPU reaches that address, satisfy the request from the
D64 image and return. This is analogous to the existing C64 KERNAL trap.

Document the exact ROM addresses used as intercept points in `c1541.c`, with a
comment citing the 1541 ROM disassembly source. These are stable across 1541
ROM versions for the addresses relevant to standard DOS operation.

## Acceptance criteria

- `via6522` timers count correctly and set IFR flags on underflow.
- CA1 interrupt fires on the correct ATN edge and sets IFR bit 1.
- `c1541_advance_one_cycle()` steps CPU and VIAs in the correct order.
- The 1541 ROM reaches its idle loop after reset (no crash during init).
- Standard D64 file loads work via the 1541 ROM when ROM is present.
- When 1541 ROM is absent, all existing KERNAL trap behaviour is unchanged.
- All existing tests pass.
- Architecture, thread ownership, and snapshot rules remain intact.
- No `src/tools/d64/` headers included from `src/machine/`.
- No SDL, Nuklear, runtime, platform, or frontend headers in `c1541.c`,
  `c1541.h`, `via6522.c`, or `via6522.h`.

## What comes next

With the 1541 running the genuine ROM, fast loaders work automatically — the
ROM executes the uploaded drive-side code natively. No fingerprint database, no
per-loader handlers. The Phase 2 fast loader work described in `C64IECPHASE_2.md`
is rendered unnecessary by this implementation.

The `C64IECPHASE_0.md` diagnostic phase (confirming the Galencia hang) remains
useful as a first step before starting implementation — it confirms the exact
failure mode and gives a known-good baseline to validate against.

Future work enabled by this foundation (all deferred, not part of this phase):
- D64 writes and `SAVE` to disk (requires VIA #1 write path).
- DOS command channel.
- Error channel.
- Additional drive ROM variants (1541-II, 1571).
- Device numbers beyond 8 and 9.
