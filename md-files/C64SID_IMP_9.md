# C64SID_IMP_9.md
# SID Improvement Phase 9 - Runtime Audio Sample Scheduling

## Component

C64MSID / runtime audio production

## Status

Coding-agent-ready implementation guide. Implement after Phase 8 has been
committed and its high-frequency finding has been recorded in `STATUS.md`.

## Purpose

Phase 8 added a physically correct one-pole output-stage low-pass filter to the
SID output path. The filter works and its tests pass, but it did not reduce the
remaining 16-22 kHz excess versus the VICE reference.

The reason is now known: the excess is not generated inside `machine/sid.c`.
It is generated by the runtime audio production schedule.

`runtime_audio_produce()` currently runs once after each runtime batch of 1024
C64 cycles. At PAL clock rate, one batch spans approximately 1.039 ms. At a
48 kHz host output rate, that batch corresponds to about 49.9 host samples. The
current code emits those samples after the batch has completed, reading the same
final `sid_sample()` value repeatedly. This is a zero-order-hold upsampling path
with an effective SID sampling cadence near:

```text
985248 Hz / 1024 cycles ~= 962 Hz
```

That creates spectral images at multiples of roughly 962 Hz, including the
16-22 kHz analysis band. Since the Phase 8 IIR runs inside SID before this
runtime-side hold, it cannot attenuate images created afterward.

This phase moves host audio sample emission to the correct cycle positions
inside the batch loop, so each emitted host sample observes a fresh SID state.

## Goal

Eliminate the runtime zero-order-hold artifact by producing host-rate audio
samples during cycle stepping rather than after a 1024-cycle batch has already
completed.

After this phase:

- SID output samples written to `audio_buffer` and to `--audio-record` observe
  updated SID state at approximately every host sample deadline.
- The runtime no longer emits dozens of identical SID samples from the last
  cycle of a batch.
- The 16-22 kHz spectral excess drops materially from the Phase 8 value of
  approximately +11.09 dB.
- No SID register-visible behavior changes.
- No platform or SDL callback behavior changes.

## In Scope

- Refactor `runtime_audio_produce()` in `src/runtime/runtime_thread.c` into a
  per-cycle or in-batch sample scheduler.
- Call the scheduler after each successful C64 cycle step while audio is
  enabled.
- Preserve existing audio recording behavior and WAV output format.
- Preserve `--audio-smoke` behavior.
- Preserve turbo mute behavior.
- Add focused tests or diagnostics proving repeated stale sample emission is
  gone.
- Run the automated SID capture comparison and record new metrics.
- Update `STATUS.md`.

## Explicit Non-Goals

Do not implement these in this phase:

- SID waveform, ADSR, filter, output gain, or output-stage changes.
- A new resampling library.
- Multi-tap interpolation, sinc resampling, FIR filtering, or oversampling.
- SDL audio callback changes.
- Platform audio buffering policy changes.
- Audio thread model changes.
- Any frontend UI change.

## Architecture Contract

Runtime may read `sid_sample(&rt->machine.sid)` and write float samples into the
shared `audio_buffer`, as established by `C64AUDFID_1` and `C64SID_IMP_1`.

The fix belongs in `src/runtime/runtime_thread.c`. The machine/SID component
continues to own SID behavior; runtime only decides when host samples are taken.

Allowed:

```text
runtime cycle loop -> c64 step -> sid advances -> runtime emits host sample
runtime emits same sample to audio buffer and optional WAV recorder
```

Forbidden:

```text
machine -> runtime
machine -> platform
runtime -> SDL or platform audio APIs
SDL audio callback -> machine or runtime
frontend -> machine
```

Do not move SID state or sample generation out of `machine/`.

## Current Failure Mode

The relevant current structure in `src/runtime/runtime_thread.c` is:

```text
for up to RUNTIME_RUN_BATCH_CYCLES:
    runtime_step_cycle(rt)
    handle trace, paste, breakpoints, frames

runtime_audio_produce(rt)
```

`runtime_audio_produce()` computes how many host samples correspond to all
cycles elapsed since the previous batch and then loops:

```text
while enough cycles for one host sample:
    sample = sid_sample(&rt->machine.sid)
    write sample
```

Because this function runs after the batch, every host sample emitted for that
batch reads the same final SID value. For a 1024-cycle batch:

```text
PAL at 48 kHz: 1024 * 48000 / 985248 ~= 49.9 identical samples
PAL at 44.1 kHz: 1024 * 44100 / 985248 ~= 45.8 identical samples
```

This is the artifact Phase 9 must remove.

## Implementation Strategy

### Refactor the Audio Helper

Replace the batch-elapsed model with a one-cycle advance model.

Suggested helper shape:

```c
static void runtime_audio_emit_sample(runtime *rt, float sample);
static void runtime_audio_advance_cycle(runtime *rt);
```

`runtime_audio_advance_cycle()` should:

1. Return immediately if there is no audio buffer and no recording path.
2. Return immediately if `audio_sample_rate <= 0`.
3. If turbo mode is active, mute by not writing samples and keep accumulator
   state from producing a burst when normal speed resumes.
4. Add `audio_sample_rate` to `audio_cycle_accum` for one C64 cycle.
5. While `audio_cycle_accum >= clock_hz`, subtract `clock_hz`, read the current
   sample, and emit it.

In pseudocode:

```c
static void runtime_audio_advance_cycle(runtime *rt) {
    uint32_t clock_hz;
    uint32_t sample_rate;
    float sample;

    if ((rt->audio_out == NULL && rt->audio_record_path == NULL) ||
        rt->audio_sample_rate <= 0) {
        return;
    }

    if (rt->speed_mode == RUNTIME_SPEED_MODE_FAST) {
        rt->audio_cycle_accum = 0.0;
        return;
    }

    clock_hz = c64_config_clock_hz(&rt->machine_config);
    sample_rate = (uint32_t)rt->audio_sample_rate;

    rt->audio_cycle_accum += (double)sample_rate;
    while (rt->audio_cycle_accum >= (double)clock_hz) {
        rt->audio_cycle_accum -= (double)clock_hz;
        if (rt->audio_smoke) {
            sample = runtime_audio_next_smoke_sample(rt, sample_rate);
        } else {
            sample = sid_sample(&rt->machine.sid);
        }
        runtime_audio_emit_sample(rt, sample);
    }
}
```

The exact helper names are not important. The important invariant is:

```text
At most one host sample deadline is handled from stale SID state; normal output
sample reads are distributed through the cycle loop after SID has advanced.
```

At PAL 48 kHz there is one host sample about every 20.53 C64 cycles, so the
`while` loop will normally emit zero samples on most cycles and one sample on
sample-deadline cycles. It should not emit large bursts under normal speed.

### Call Site

Call the per-cycle helper after `runtime_step_cycle(rt)` succeeds and after SID
has advanced for that cycle.

The natural location is inside the existing run loop immediately after:

```c
if (!runtime_step_cycle(rt)) {
    break;
}
```

Add:

```c
runtime_audio_advance_cycle(rt);
```

Remove the existing post-batch call:

```c
runtime_audio_produce(rt);
```

or leave a renamed wrapper only if it no longer emits batch-held samples.

### Reset Behavior

`runtime_audio_reset()` should continue to:

- reset the shared audio buffer;
- clear `audio_cycle_accum`;
- clear smoke phase;
- finish/reopen recording as it does today;
- call `runtime_update_sid_sample_output(rt)`.

`audio_last_cycle` may become unnecessary after this refactor. If it is removed,
remove it from `runtime_internal.h` and all reset/update code. If it is kept for
diagnostics, do not use it to batch-produce stale samples.

### Pause, Step, and Breakpoints

The per-cycle helper should be called only for cycles that actually executed.
This preserves recording duration in terms of emulated time, not wall-clock time.

If a breakpoint pauses execution after a cycle, the sample for that completed
cycle may already have been emitted. That is acceptable and consistent with the
machine having advanced.

Single-cycle and instruction stepping may produce audio if audio recording or
playback is enabled. This is acceptable, but tests should confirm it does not
write bursts.

### Smoke Tone

`--audio-smoke` should continue to prove the runtime/platform audio path without
requiring SID output. It can use the same per-cycle sample deadlines and update
its 440 Hz phase only when a host sample is emitted.

Do not enable SID sample output for smoke mode; `runtime_update_sid_sample_output`
currently disables SID output when `audio_smoke != 0`, and that behavior should
remain.

### Turbo Mode

Turbo mode should remain muted to avoid flooding the buffer.

When turbo mode is active, do not accumulate unlimited pending audio. Either:

- keep `audio_cycle_accum` at 0.0 while turbo is active, or
- clamp/discard pending accumulator state when entering turbo.

The important behavior is that returning from turbo must not emit a delayed
burst of old samples.

## Suggested Tests

Add focused coverage where it is least invasive. If exposing runtime internals
to tests would require broad refactoring, prefer a small test-only helper or a
local diagnostic script, but do not skip verification entirely.

Suggested automated tests:

- `test_runtime_audio_per_cycle_no_batch_hold` - run a tiny runtime/machine
  cycle loop with audio recording or a test buffer enabled. Feed changing SID
  output and verify a produced group of host samples is not a long run of
  identical values caused by one batch-end sample.

- `test_runtime_audio_sample_count_matches_cycles` - for a known PAL cycle
  count and sample rate, verify emitted sample count matches the fractional
  accumulator expectation within one sample.

- `test_runtime_audio_turbo_discards_pending_audio` - enter turbo mode for many
  cycles and return to normal speed; verify the audio buffer does not receive a
  burst proportional to the skipped turbo cycles.

- `test_runtime_audio_recording_uses_same_samples_as_buffer` - when both the
  shared buffer and recorder are enabled, verify both paths receive the same
  emitted sample sequence.

- Existing `tests/machine/test_sid.c` tests must remain at 60 passing tests.
  Phase 9 should not require changing SID tests except for build fallout from
  renamed runtime helpers.

If adding runtime tests is too large for this phase, add a narrow diagnostic
mode or script that records consecutive emitted samples for a short run and
asserts that maximum identical-run length is far below the old 45-50 sample
batch-hold length.

## Key Metrics To Record

Compare Phase 9 against the Phase 8 baseline:

- `spectral.band_16_22_khz_db` - should drop materially from the Phase 8
  excess of approximately +11.09 dB.
- `spectral.mae_db` - should improve from the Phase 8/7A range.
- `rms` - should remain within 5% of the VICE reference for the same comparison
  file.
- `alignment.correlation` - should remain near the Phase 7A/8 baseline unless
  the improved sampling reveals a real timing issue.
- `score` - should improve, or any regression must be explained by the metric
  breakdown.
- Audio buffer overrun/underrun counters - should not show a new steady-state
  regression during normal playback.

Expected outcome:

```text
The high-frequency excess caused by 1024-cycle zero-order hold should fall.
The exact score improvement is not known before measurement, but the 16-22 kHz
band should no longer be dominated by images of a ~962 Hz sample hold.
```

## Acceptance Criteria

- Existing tests pass.
- New Phase 9 runtime audio tests or diagnostics pass.
- `runtime_audio_produce()` no longer emits many host samples from one
  batch-final SID value.
- Runtime audio sample emission is tied to cycle stepping and host sample
  deadlines.
- `--audio-record` still writes a valid mono WAV file of the requested duration.
- `--audio-smoke` still produces a bounded 440 Hz square wave.
- Turbo mode remains muted and does not produce a catch-up burst.
- Normal playback does not introduce obvious buffer underrun/overrun regressions.
- The 16-22 kHz band excess improves materially versus Phase 8, or a new
  diagnosis is recorded if it does not.
- `STATUS.md` is updated only after implementation and tests support the claim.
