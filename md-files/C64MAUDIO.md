# C64MAUDIO.md

# Audio Output Infrastructure Plan for c64m

## Purpose

This document defines the high-level plan for getting generated C64 audio from
the emulation thread to the host audio device without violating the c64m
architecture rules.

This is not a coding-agent-ready implementation guide. It is the planning level
needed before writing a detailed audio infrastructure specification.

## Goal

Provide a host audio output path suitable for SID playback.

The audio path must:

```text
- let runtime write generated SID samples;
- let the SDL audio callback read samples;
- avoid runtime -> platform dependencies;
- avoid machine -> platform dependencies;
- avoid live machine access from the audio callback;
- tolerate ordinary host scheduling jitter;
- fail safely on underrun or overrun.
```

## Architecture Boundary

The required ownership model is:

```text
machine/SID:
    owns SID register state and SID sample generation behavior

runtime:
    steps the machine and writes generated audio samples

util:
    owns the generic audio sample buffer type

platform:
    owns SDL audio device creation and callback

frontend:
    owns user-facing audio configuration UI, if added
```

The runtime may write into a `util/` audio buffer. The SDL callback may read from
that same buffer. Runtime must not include platform headers or call SDL.

## Thread Model

```text
Emulation thread:
    produces samples
    writes samples to the audio buffer

SDL audio callback thread:
    reads samples from the audio buffer
    fills the host stream

UI thread:
    may configure audio through normal runtime/platform setup paths
    must not touch live SID state
```

The audio callback must never block on the emulation thread.

## Audio Buffer Direction

Use a single-producer/single-consumer design unless the detailed guide justifies
something else.

```text
producer: runtime/emulation thread
consumer: SDL audio callback thread
```

The buffer should store interleaved host samples or emulator-native samples. The
detailed guide must choose one and document it.

Recommended starting point:

```text
sample format: signed 16-bit or float32, chosen once for platform output
channels: mono generated by SID, duplicated to stereo only at platform boundary
host rate: 48000 Hz unless SDL selects another rate
```

## Resampling Policy

The C64 machine clock and host audio rate are not the same.

The detailed guide must define:

```text
- PAL machine clock to host sample conversion;
- NTSC machine clock to host sample conversion;
- sample accumulator or fractional stepping strategy;
- whether SID emits at host sample rate or at an intermediate rate;
- behavior during turbo mode, pause, reset, and speed changes.
```

Recommended initial policy:

```text
- generate SID samples at the host audio rate using a fractional machine-cycle
  accumulator;
- keep the SID internal phase at machine-cycle precision where practical;
- output silence while paused unless a later feature explicitly needs held audio.
```

## Underrun and Overrun Policy

The implementation guide must define exact behavior.

Suggested policy:

```text
underrun:
    output silence for missing samples and count the underrun

overrun:
    drop oldest samples or reject new samples, but choose one policy and count it
```

Underruns and overruns should be observable through logging or optional debugger
status, but should not spam logs every callback.

## Runtime Integration

Runtime should produce audio in regular batches. Candidate integration points:

```text
- every N machine cycles;
- every raster line;
- every frame;
- every fixed host-sample batch.
```

The detailed guide should prefer a low-latency batch size that is still simple
and stable. Frame-sized audio batches are simple but may be too coarse for latency
and underrun behavior.

## Platform Integration

Platform should:

```text
- initialize SDL audio;
- request the chosen sample format and rate;
- expose the actual accepted format/rate to setup code;
- own the SDL callback;
- read from the shared audio buffer in the callback;
- fill missing samples with silence.
```

Platform must not know SID registers or C64 timing semantics.

## Frontend Integration

Frontend audio UI is optional for the first implementation. If added, it should
control only user-facing settings such as:

```text
- audio enabled/disabled;
- output volume;
- buffer size or latency preset;
- mute.
```

Frontend must send changes through normal command/config paths.

## Acceptance Criteria for a Future Implementation Guide

A detailed implementation guide should define tests or smoke checks for:

```text
- audio device opens successfully;
- runtime can write samples without platform dependency;
- SDL callback can read samples without touching runtime or machine;
- underrun produces silence, not crashes;
- overrun follows the documented policy;
- PAL and NTSC sample generation remain stable;
- reset and pause do not leave stale unsafe audio state;
- existing video, debugger, keyboard, PRG, and D64 behavior still works.
```

## Explicit Non-Goals

```text
- SID emulation details. Those belong in C64MSID.md.
- Music playback UI.
- Recording WAV files.
- MIDI.
- Audio effects.
- Perfect latency tuning.
- Platform-specific Core Audio, WASAPI, or ALSA backends outside SDL.
```
